# OpenCode Feature Inventory

Last updated: 2026-07-07.

This inventory uses official OpenCode docs as a capability baseline, not an implementation clone target. Each capability maps to a hya-native verification owner so capability work becomes executable behavior, not a checklist.

Priority vocabulary: `must-have` blocks the Real coding agent target; `should-have` blocks competitiveness but not the initial verification architecture; `nice-to-have` waits; `out-of-scope` is deliberately excluded.

hya-native coverage is first-pass and sourced from current project docs: the compatibility tracker and `docs/hya-pi-compat-comparison.md`.

## hya-native extension capabilities

| Capability family | Priority | hya behavior target | OpenCode analogue/source | Current hya-native coverage | Verification owner |
| --- | --- | --- | --- | --- | --- |
| Verification bootstrap | must-have | Product behavior is defined by boundary-owned suites, starting with Registered tool contracts, then Golden path and TUI verification. | OpenCode exposes the capability baseline; verification design is hya-native. | missing as a complete suite; scattered tests exist. | Registered tool contract; Golden path; TUI verification suite |
| Rust plugin binary | must-have | Private plugins are Rust executables speaking hya's stable plugin protocol, distinct from JS/TS OpenCode plugins and in-process dynamic libraries. | [Plugins](https://opencode.ai/docs/plugins/), [Custom tools](https://opencode.ai/docs/custom-tools/) | partial: stdio plugin host and plugin tools exist; private Rust distribution story needs contract tests/docs. | `hya-plugin` protocol suite; plugin tool contract |
| Runtime plugin registration | must-have | Add/start plugin capabilities while the runtime is already running; next admitted Turn sees the updated tool catalog. | OpenCode loads plugins/tools from config/startup; hya extends this with runtime registration. | missing/partial: dynamic HTTP MCP exists in places; plugin runtime registration not complete. | Plugin lifecycle boundary-owned suite |
| Hot plugin reload | should-have | Replace a registered plugin instance without restarting runtime; visibility follows ADR 0008 next-turn tool visibility. | [Plugins](https://opencode.ai/docs/plugins/) | missing. | Plugin lifecycle suite; event/projection assertions |
| Lazy plugin discovery | should-have | Defer configured plugin startup until its capabilities are needed. | [Plugins](https://opencode.ai/docs/plugins/), [Config](https://opencode.ai/docs/config/) | missing. | Plugin lifecycle suite |
| Runtime skill registration | must-have | Add skill definitions to a running runtime for subsequent Turns/new Sessions. | [Skills](https://opencode.ai/docs/skills/) | missing/partial: static discovery and skill tool exist. | Skill plane suite |
| Hot skill reload | should-have | Refresh a registered skill without runtime restart; visibility follows ADR 0007 next-turn visibility. | [Skills](https://opencode.ai/docs/skills/) | missing. | Skill plane suite; prompt snapshot tests |
| Lazy skill discovery | should-have | Defer catalog expansion until a skill name/search requires it. | [Skills](https://opencode.ai/docs/skills/) | missing. | Skill plane suite |
| Team-grade multi-agent management | must-have | Better live Team/Roster/mail/subagent management than OpenCode baseline, with separate child Sessions and observable subagent state. | [Agents](https://opencode.ai/docs/agents/), [TUI](https://opencode.ai/docs/tui/) | partial: runtime primitives and TUI subagent visibility exist; management UX and contracts need hardening. | `hya-core` team suite; TUI verification suite |
| Performance envelope | must-have | High-performance Rust runtime must keep tool calls, event replay, TUI projection, and provider streaming efficient under real sessions. | OpenCode docs do not define a performance contract. | undefined as testable gates. | Bench/smoke suite after behavior contracts |

## Bootstrap order

1. Write Registered tool contracts for canonical `hya-tool` built-ins and permission behavior.
2. Add the Golden path: prompt -> built-in tool -> persisted event log -> projection -> TUI render.
3. Add TUI rendering contracts/snapshots/interaction tests for command, permission, question, session, and subagent surfaces.
4. Add plugin and skill lifecycle suites for runtime registration first, then hot reload, then lazy discovery.
5. Expand provider, MCP, CLI/server, snapshot, and share/import/export suites as the inventory rows move from partial to exists.
