# Session naming and persistence design

> **Superseded note:** ADR-0005 removed the backend legacy TUI and JSON history surface after this plan. Treat `crates/hya-backend/src/tui*` hooks and legacy `HistoryStore` references below as historical context; current work should use `crates/hya`/`crates/hya-tui` frontend seams plus server/session APIs.


## Goal

Make hya sessions durable, recoverable, and user-readable while preserving the event-sourced architecture. New hya sessions use `hysec_` IDs, the SQLite event log remains the source of truth, and session lists/switchers show names/titles instead of raw IDs.

## Non-goals

- Do not rewrite historical `event_log` rows.
- Do not make SQL materialized session/message tables authoritative in this task.
- Do not remove legacy JSON history in one step; bridge away from it and stop relying on it for current session lists.
- Do not add a denormalized session index/cache unless tests or profiling prove the projection-backed list path is inadequate.
- Do not change `MessageId`, `PartId`, `AgentName`, or other UUID/newtype IDs unless a compiler error proves the session-only change cannot be isolated.

## Source-backed constraints

- `SessionEngine::emit` currently persists each event with `SessionStore::append_event`, then dispatches hooks and publishes the envelope to `EventBus`.
- `SessionStore` currently stores `event_log.session_id` and `token_ledger.session_id` as UUID bytes derived from `SessionId::as_uuid()`.
- Compat-compatible server session lists already replay SQLite events through `SessionStore::list_sessions` and `load_session`.
- Backend TUI switching/resume still uses `HistoryStore` JSON metadata under `~/.hya/history` and must be bridged to SQLite to remove split-brain behavior.
- Upstream Compat default persistent titles are `New session - <ISO8601 UTC>` / `Child session - <ISO8601 UTC>`, and auto-title generation runs only for root sessions that still have a default title and exactly one real user turn.
- The user requirement intentionally diverges from upstream empty-session handling: hya must discard created-but-unused unnamed sessions on exit/list finalization, and non-empty unnamed hya sessions must show `Untitled Session_<year>-<month>-<date>-<hour>-<minutes>` until Compat-compatible title generation replaces it.

## Data contracts

### Session ID grammar

New primary hya session IDs:

```text
hysec_[A-Za-z0-9]{20}
```

Generation requirements:

- Prefix is exactly `hysec_`.
- Suffix length is exactly 20 Unicode scalar values, all ASCII alphanumeric.
- Suffix is produced from OS-backed cryptographic randomness through `rand`/`rand_core`; it is not derived from timestamps, UUIDs, counters, model text, or process IDs.
- The parser accepts only ASCII alphanumeric suffix characters for `hysec_` IDs.

Compatibility read grammar:

```text
hysec_[A-Za-z0-9]{20}      # new primary ID
ses_[0-9a-fA-F]{32}        # legacy hya/Compat-style UUID-simple display
[0-9a-fA-F-]{36}           # legacy raw UUID
[0-9a-fA-F]{32}            # legacy UUID-simple, only if current parser already accepts it or tests require it
```

Emit contract:

- Newly created sessions emit/display `hysec_...` immediately across proto, server, CLI/TUI, and client URLs.
- Legacy sessions loaded from old SQLite rows may continue to display as `ses_<uuid-simple>` unless a future migration creates explicit aliases. This task must at least keep them readable/listable/deletable.

Storage key contract:

- Add a `SessionId::storage_key()` method or equivalent shared function owned by `hya-proto` or `hya-store`.
- For new `hysec_` IDs, `event_log.session_id` and `token_ledger.session_id` use the ASCII ID bytes, including the `hysec_` prefix.
- For legacy UUID-backed IDs, storage keys remain the existing 16 UUID bytes.
- `SessionStore::list_sessions` must decode both storage key shapes. Unknown key shapes are skipped with a bounded warning only if existing store error policy supports logging; otherwise they remain skipped as current UUID parse failures are skipped.
- Event payloads serialize `SessionId` in the same display form users/API clients see.

This is additive and avoids historical event-log rewrites.

### Title contract

Title states:

1. **Default/fallback**: generated by hya when a session is non-empty but no model title exists yet.
2. **Auto**: generated by Compat-compatible title logic after the first real root user turn.
3. **Manual**: set by explicit update/rename routes; never overwritten by auto-title.

Canonical hya fallback title:

```text
Untitled Session_%Y-%m-%d-%H-%M
```

Time behavior:

- Use UTC for deterministic tests and crash recovery.
- Format fields are four-digit year, two-digit month, two-digit day, two-digit 24-hour hour, two-digit minute.
- The timestamp comes from the first durable non-empty session activity timestamp, not wall-clock time at list rendering.

Compat-compatible auto-title behavior:

- Only root sessions are eligible.
- Only sessions still on a default/fallback title are eligible.
- Exactly one real, non-synthetic user turn must exist.
- Use the same title-agent prompt constraints as Compat: same language as user request when possible, single line, concise title, no surrounding quotes.
- Strip `<think>...</think>` blocks from the title result.
- Store the first non-empty line.
- Truncate stored auto-title to 100 characters, using `...` when truncation occurs.
- Auto-title writes use `SessionTitled` through `SessionEngine::set_title` or a shared title helper, so projection remains authoritative.

Manual title behavior:

- Existing PATCH/update title routes remain the manual rename path.
- A manual title is sticky. The auto-title trigger must check projection/title origin and not overwrite manual titles.
- If the existing event model cannot encode title origin directly without a broad event schema change, use conservative inference in this task: auto-title only if the current title matches the hya fallback pattern or Compat default-title pattern. Manual titles that happen to match those patterns are a known limitation unless title-origin metadata is introduced.

### Empty session visibility

Definition:

- Empty session: projection has `SessionCreated` and no real user/assistant interaction and no manual/generated title.
- Non-empty session: projection has at least one real user or assistant message, or a manual title.

Behavior:

- New empty sessions may be created durably so direct API lookups and internal handles are stable.
- List/switch/resume surfaces hide empty unnamed sessions.
- Exit/finalization deletes an empty unnamed session from SQLite where a concrete exit hook exists; list filtering is the required safety net for crash paths where cleanup cannot run.
- Direct GET by ID for an empty session may return the session while the process is alive; after successful cleanup it returns not found.

Concrete cleanup ownership:

- `hya-core` owns the reusable cleanup helper, e.g. `SessionEngine::cleanup_empty_unnamed_session(session) -> Result<bool, CoreError>`. The helper reads the session projection, applies the shared empty-unnamed predicate, and calls `SessionEngine::delete_session` only when the predicate is true. This keeps server tests, backend TUI, and any future runtime from depending on a backend-only helper.
- The backend TUI has a concrete async finalization hook in `crates/hya-backend/src/tui.rs::run` after the event loop stops, pending prompts are cancelled, and the active turn is cancelled/joined. This path owns both the current `SessionId` and `SessionEngine`, so it must call the core cleanup helper.
- Backend TUI current-session abandon paths (`TuiEffect::NewSession` and `TuiEffect::ResumeSession`) must call the same core cleanup helper before replacing the current `session` value.
- Compat/server API create paths do not have a per-session process-exit hook. They must hide empty unnamed sessions in list/search/switch surfaces and keep explicit delete semantics; do not add broad server process shutdown cleanup without a concrete per-session owner.
- Do not implement cleanup in synchronous `Drop`, panic-hook, or terminal-restore code because those paths cannot reliably await store/projection operations.

### Streaming durability contract

Current engine behavior persists every stream delta. New behavior must split live publication from durable persistence.

Required semantics:

- Live TUI/SSE subscribers still receive streaming assistant deltas while the model is producing output.
- Durable replay after successful assistant completion contains the final assistant text.
- Token-by-token assistant `TextDelta` events are not individually required to be durably committed.
- If the process crashes during an assistant stream before completion, partial assistant text may be lost from durable replay. The already-admitted user prompt and prior completed messages remain recoverable.

Implementation seam:

- Add an internal publishing path, e.g. `publish_live(session, event)`, that builds an `Envelope` for hooks/bus without `SessionStore::append_event`.
- Keep `emit(session, event)` as the durable append + publish path.
- In assistant streaming code, buffer text deltas per part/message, publish live deltas, and durably emit a coherent final sequence at stream end: assistant message start if not already durable, text start, final text delta or replacement-compatible event shape, text end, and message finish.
- If current projection only supports `TextDelta`, the durable final text can be one `TextDelta` containing the full accumulated text followed by `TextEnd`; do not add a new event variant unless tests show the current reducer cannot represent final text this way.
- Tool events and non-text assistant parts need explicit treatment before code: either continue durable immediate writes for non-text events or buffer them with the assistant message. Do not silently drop tool-call state.

## Cross-layer flow

```text
SessionId::new()
  -> hya-core SessionEngine::create/create_with_id
  -> Event::SessionCreated payload
  -> hya-store storage_key(event_log.session_id)
  -> SessionStore::replay/read_projection/list_sessions
  -> hya-server Compat/native routes
  -> hya-sdk/hya-tui/backend TUI switch/list surfaces
```

The only owner for ID parsing/rendering is `hya-proto::SessionId` plus the store storage-key helper. Server, client, and TUI code must call that shared parser/display and must not duplicate prefix validation.

```text
First real user prompt
  -> durable admission events
  -> projection marks non-empty
  -> fallback title if no title exists
  -> Compat-compatible title trigger if eligible
  -> SessionTitled event
  -> list/switch/search uses projection title
```

The only owner for fallback/default/title eligibility should be a shared Rust module close to the event/projection boundary, preferably in `hya-core` if it needs projection and model summarizer access, or `hya-proto` if it is pure formatting/detection.

## Implementation boundaries by crate

### `crates/hya-proto`

- Replace session-only UUID storage with a representation that can hold `hysec_` and legacy UUID IDs.
- Preserve legacy parse/display for old sessions.
- Add tests for ID grammar, generation, parse rejection, storage-key round trip, and serde round trip.
- Keep other ID types unchanged.

### `crates/hya-store`

- Replace `session.as_uuid().as_bytes()` usage with shared storage-key encoding.
- Decode both ASCII `hysec_...` and legacy UUID-byte keys in `list_sessions`.
- Keep delete/replay/read_usage using the same storage key for the parsed `SessionId`.
- Add tests using both legacy and new key shapes.

### `crates/hya-core`

- Keep event-sourced writes through `SessionTitled` and message/text events.
- Add live-only publish seam for streaming.
- Add or reuse title-generation trigger with Compat-compatible gates.
- Add empty-session cleanup/list-filter helper based on projection.

### `crates/hya-server`

- Update native and Compat route parsing to accept `hysec_` through `SessionId::from_str` only.
- Replace first-prompt truncation with the shared title trigger.
- Ensure list/search filters use projection title and hide empty unnamed sessions.
- Keep archived/root/search/cursor behavior unchanged except for the hidden-empty filter.

### `crates/hya-backend` / `crates/hya-tui` / `crates/hya-sdk`

- Replace raw UUID URL formatting (`as_uuid`) with `SessionId::to_string()` where URLs take session IDs.
- Bridge session switch/resume lists to SQLite-backed summaries. During bridge, DB summaries win over JSON history by session ID; JSON can be an import source but not the current source of truth.
- Remove display fallback that uses assistant stream text as a pseudo-title.
- Active session labels should prefer title/fallback, with UID only as internal value or secondary debug text.

## Migration and rollback

- No destructive schema migration in this task.
- Existing SQLite rows with 16-byte session keys continue to list/replay/delete.
- New SQLite rows use ASCII `hysec_...` keys.
- JSON history import, if touched, must be idempotent. Re-running import must not duplicate events.
- Rollback to a pre-change binary may not understand new `hysec_` ASCII keys. That is acceptable only if documented in release notes; otherwise gate the write flip behind a config flag. Since the user explicitly requires new `hysec_` IDs, this task writes new IDs by default and documents rollback as “old binaries can read old sessions, not new hysec sessions.”

## Risks and mitigations

| Risk | Mitigation |
| --- | --- |
| Split-brain between SQLite and JSON history | Make SQLite projection the list/switch source; JSON only bridge/import. |
| ID parsing duplicated in routes | Centralize parser in `SessionId::from_str`; tests cover all routes. |
| Legacy sessions disappear | Decode legacy 16-byte keys in store list/replay/delete tests. |
| Manual titles overwritten | Gate auto-title to fallback/default patterns only, or add explicit title-origin event if needed. |
| Streaming deltas vanish from live UI | Live-only publish path covered by an engine/bus test before durable write change. |
| Crash creates empty ghosts | Hide empty unnamed sessions in list/switch; cleanup on known exit path. |
| Event-log/reducer drift | Keep projection-derived titles/messages; no authoritative SQL side table. |

## Acceptance mapping

- `hysec_` ID generation/parsing: `hya-proto` unit/property tests.
- SQLite round trip for new and legacy sessions: `hya-store` integration tests.
- Empty unnamed hidden/deleted: store/server list tests plus backend switch summary tests.
- Fallback title format: pure title-format tests with fixed UTC timestamp.
- Compat title replacement: server prompt/title tests with fake summarizer/title agent where existing test scaffolding allows it.
- Switch/session display names: backend/TUI summary tests and SDK/server list tests.
- Streaming live vs durable behavior: core engine/store test proving live bus receives deltas while replay receives final completed content.
